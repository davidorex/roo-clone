# Code Parsing and Analysis Specifics in `src/services/code-index`

The core of code analysis within the `src/services/code-index` module lies in the `CodeParser` class (found in `processors/parser.ts`). This parser is responsible for taking raw source code files and transforming them into structured units called `CodeBlock` objects. These blocks are the fundamental pieces of information used for generating embeddings and enabling semantic search.

## The `CodeBlock` Structure

A `CodeBlock` represents a semantically relevant segment of code extracted from a file. Its structure is defined as follows:

```typescript
interface CodeBlock {
	file_path: string // Absolute path to the source file
	identifier: string | null // Name of the primary entity (e.g., function name, class name)
	type: string // Type of the code structure (e.g., "function_definition", "class_declaration")
	start_line: number // 1-based starting line number of the block in the file
	end_line: number // 1-based ending line number of the block in the file
	content: string // The actual text content of the code block
	segmentHash: string // SHA256 hash of this specific code block (filePath + lines + content)
	fileHash: string // SHA256 hash of the entire content of the source file
}
```

## How `CodeBlock` Fields Are Populated

The `CodeParser` utilizes the `tree-sitter` parsing library to analyze code. `tree-sitter` builds an Abstract Syntax Tree (AST) for the source file, allowing for precise identification of code structures.

1.  **`file_path`**: This is the absolute path of the file being parsed. It's provided as input to the `parseFile` method.

2.  **`content` (of the block)**:

    - For blocks derived from `tree-sitter` captures (the primary method), this is the exact text content of the captured AST node (e.g., the full text of a function definition).
    - For blocks generated by fallback or line-based chunking (when `tree-sitter` structures are too large or not found), this is the concatenated text of the lines forming that chunk.

3.  **`start_line` and `end_line`**:

    - These are 1-based line numbers indicating the span of the `CodeBlock` within the original file.
    - For `tree-sitter` derived blocks, these come directly from the `startPosition.row` and `endPosition.row` of the AST node.

4.  **`type`**:

    - For `tree-sitter` derived blocks, this is the `type` string of the AST node itself (e.g., `"function_definition"`, `"class_declaration"`, `"method_definition"`). The specific types depend on the `tree-sitter` grammar for the language being parsed.
    - For blocks from fallback chunking, the type is typically `"fallback_chunk"`.
    - For oversized lines that are split, the type might be `"{original_node_type}_segment"`.

5.  **`identifier`**:

    - The parser attempts to extract a meaningful name for the code block.
    - It looks for specific child nodes within the captured AST node that typically hold names, such as a node with the field name `"name"` or a child node of type `"identifier"`.
    - If no such identifier can be clearly determined from the AST structure, this field will be `null`. This is common for chunks generated by fallback or line-based splitting of larger, non-identifiable text segments.

6.  **`fileHash`**:

    - Before parsing, the `CodeParser` calculates a SHA256 hash of the entire content of the source file. This hash is associated with every `CodeBlock` generated from that file.
    - It's used by the `CacheManager` to detect if a file has changed since the last indexing operation, avoiding redundant parsing and embedding.

7.  **`segmentHash`**:
    - This is a SHA256 hash generated uniquely for each `CodeBlock`.
    - It's typically created by hashing a combination of the `file_path`, `start_line`, `end_line`, and the block's `content`.
    - This hash helps in uniquely identifying individual code segments and can be used to prevent duplicate processing or storage if the exact same segment appears multiple times (though the primary ID for Qdrant points is generated differently, based on file path and start line).

## Role of Tree-sitter

- **Language-Specific Parsing**: `CodeParser` dynamically loads `tree-sitter` grammars for various programming languages (defined in `scannerExtensions`). This allows it to understand the syntax and structure of different code types.
- **Structural Identification**: Pre-defined `tree-sitter` queries are used to pinpoint specific, semantically relevant nodes in the AST (like functions, classes, methods, interfaces, etc.). These captures are the primary source for creating `CodeBlock`s.
- **Metadata Extraction**: Information like node type, start/end positions, and potential identifiers is directly extracted from the `tree-sitter` AST nodes.

## Supported Languages

The range of languages that can be parsed and analyzed is determined by the available `tree-sitter` grammars configured in the system and the list of file extensions in `src/services/code-index/shared/supported-extensions.ts`.

## Output of Analysis

The result of the parsing process is an array of `CodeBlock` objects. Each `CodeBlock` is a self-contained unit representing a chunk of code with associated metadata. These blocks are then passed on to the embedding generation stage of the indexing pipeline. The quality and granularity of these blocks, as determined by the parsing and chunking heuristics (detailed in `src_services_code_index_chunking_heuristic.md`), directly impact the effectiveness of the semantic search.

## Handling of API Contracts and Dependencies

The system's analysis of API contracts (like interface definitions, type aliases, function signatures) and dependencies (like import/export statements) is based on their textual representation within the code blocks:

- **Textual Embedding**: If an API contract or a dependency statement is part of the text content of a `CodeBlock` (either because `tree-sitter` identified it as a distinct unit or it's part of a larger chunk), its textual form will be included in `CodeBlock.content` and subsequently embedded as a vector. The `CodeBlock.type` will reflect the `tree-sitter` node type (e.g., `"interface_declaration"`, `"import_statement"`).
- **No Specialized Structured Extraction**: The system **does not** perform a deeper, specialized analysis to extract structured data _about_ these elements. For instance:
    - It doesn't parse out individual method names, parameters, and return types from an interface definition and store them as separate, queryable metadata fields in Qdrant.
    - It doesn't build a dependency graph or store a list of imported/exported modules as distinct, filterable fields in the Qdrant payload.
- **Semantic Understanding via Embeddings**: The "understanding" of these contracts and dependencies comes from the semantic meaning captured by the vector embedding of their textual content. Searches for related concepts will rely on the similarity of these embeddings.

Therefore, while the text of API contracts and dependencies is processed and vectorized if it forms part of a code chunk, the system does not create a structured, relational database of these specific code intelligence aspects.
