# Removing "Modes" and "Starting Tasks" Notices from System Prompt

This document outlines the plan to remove information about (1) available modes and mode switching/creation, and (2) the AI's ability to start new tasks, from the system prompt. This aligns with a stricter user-directive model where the AI is more passive and only acts on explicit user commands.

## Background

The current system prompt generation process, primarily orchestrated by `src/core/prompts/system.ts`, includes:

1.  **Modes Information**:

    - A dedicated "MODES" section is generated by the `getModesSection` function (located in `src/core/prompts/sections/modes.ts`). This section lists all available modes, provides brief descriptions, and includes instructions for using the `fetch_instructions` tool with the `create_mode` task to guide users in creating or editing modes. This entire section is then incorporated into the main system prompt.

2.  **Starting Tasks Information**:
    - The `new_task` tool, which allows the AI to initiate a new task, has its description included in the "# Tools" section of the prompt. These tool descriptions are assembled by the `getToolDescriptionsForMode` function (in `src/core/prompts/tools/index.ts`).

Removing these elements will reduce the AI's awareness of these meta-capabilities, reinforcing its role as a direct responder to user directives.

## Plan for Removal

### Part 1: Removing "Modes" Notices from the System Prompt

The most direct way to remove all "Modes" notices is to prevent the "MODES" section from being generated and included.

1.  **Modify `src/core/prompts/system.ts`**:

    - **Target File**: `src/core/prompts/system.ts`
    - **Target Function**: `generatePrompt` (internal function called by the exported `SYSTEM_PROMPT`)
    - **Action**:
        - Locate the part where `getModesSection(context)` is called. This is typically done within a `Promise.all` if other asynchronous sections are fetched simultaneously. The result is usually stored in a variable (e.g., `modesSection`).
            ```typescript
            // Current conceptual structure in generatePrompt:
            // const [modesSection, mcpServersSection] = await Promise.all([
            //  getModesSection(context), // Call to generate the MODES section
            //  // ... other async calls ...
            // ]);
            ```
        - Modify this to remove the call to `getModesSection`. If it was part of `Promise.all`, adjust the destructuring accordingly. Ensure that the variable `modesSection` is either not declared or is assigned an empty string.
            ```typescript
            // Modified conceptual structure in generatePrompt:
            // const modesSection = ""; // Assign empty string or remove declaration
            // const mcpServersSection = await getMcpServersSection(...); // Handle other calls separately if needed
            ```
        - In the `basePrompt` template string within `generatePrompt`, remove the placeholder that includes the `modesSection` content.
            ```typescript
            // In the basePrompt template string, find and remove:
            // ${modesSection}
            ```
    - **Impact**: This change will completely prevent the "MODES" section (listing available modes and how to create/edit them) from being part of the system prompt. The AI will no longer be explicitly informed about the concept of different modes or how to manage them via `fetch_instructions`.

2.  **Consider the `switch_mode` Tool (Further Refinement - Optional)**:
    - Even after removing the "MODES" section, the `switch_mode` tool's description might still appear in the "# Tools" section if the tool is enabled for the current operational mode.
    - If the objective is to entirely hide the concept of modes from the AI, the `switch_mode` tool description would also need to be prevented from appearing. This would involve a modification similar to that described in Part 2 for the `new_task` tool, by filtering `switch_mode` out in `src/core/prompts/tools/index.ts`. This is an optional step depending on how strictly the "no modes notice" rule should be applied.

### Part 2: Removing "Starting Tasks" (new_task tool) Notices from the System Prompt

The primary way the AI is informed about its ability to start new tasks is through the description of the `new_task` tool.

1.  **Modify `src/core/prompts/tools/index.ts`**:

    - **Target File**: `src/core/prompts/tools/index.ts`
    - **Target Function**: `getToolDescriptionsForMode`
    - **Action**:

        - Inside `getToolDescriptionsForMode`, a `Set` named `tools` is populated with all tools applicable to the current mode.
        - Before this `Set` is iterated to generate descriptions, explicitly delete the `new_task` tool from it.

            ```typescript
            // In getToolDescriptionsForMode, after the 'tools' Set is fully populated
            // (i.e., after adding tools from mode's groups and ALWAYS_AVAILABLE_TOOLS):

            tools.delete("new_task") // Add this line to remove new_task from the set

            // The subsequent code that maps over 'tools' to get descriptions will now exclude 'new_task'.
            // const descriptions = Array.from(tools).map((toolName) => { /* ... */ });
            ```

    - **Impact**: This ensures that the `new_task` tool's description is not included in the "# Tools" section of the system prompt. The AI will not be explicitly informed of a tool to initiate new tasks.

2.  **Review `src/core/prompts/sections/objective.ts` (Minor Textual Consideration - Optional)**:
    - **Target File**: `src/core/prompts/sections/objective.ts`
    - **Target Function**: `getObjectiveSection`
    - **Action**:
        - The current "OBJECTIVE" section includes the statement: "1. Analyze the user's task and set clear, achievable goals to accomplish it."
        - While this doesn't directly mention _starting new_ tasks, in the context of a very strict user-directive model, "Analyze the user's task" might imply more initiative than desired.
        - A subtle rephrasing could be considered to emphasize responsiveness to the _current directive_.
        - **Example Optional Rephrasing**:
            > "1. Carefully understand the user's current directive and identify the immediate goal(s) required to fulfill it."
        - This is a minor change and its necessity depends on the desired strictness. The removal of the `new_task` tool description is the more direct and impactful change for preventing AI-initiated tasks.

## Verification Steps

After implementing these changes:

1.  **Compile**: Ensure the application compiles without any errors.
2.  **Inspect System Prompt**:
    - Thoroughly check the generated system prompt.
    - Confirm that the entire "MODES" section (which lists modes and mentions `fetch_instructions` for `create_mode`) is completely absent.
    - Confirm that the `new_task` tool is not listed under the "# Tools" section.
    - If the "OBJECTIVE" section was rephrased, verify the new wording.
3.  **Behavioral Observation (Indirect)**:
    - Observe the AI's behavior. It should not spontaneously discuss different modes, suggest switching modes (unless the `switch_mode` tool itself was also removed/hidden), or offer to start new, independent tasks. Its actions should be strictly in response to direct user commands.

By applying these modifications, the system prompt will be more aligned with a model where the AI is a passive agent, waiting for and acting only upon explicit user directives, without awareness of mode management or self-initiated task creation.
